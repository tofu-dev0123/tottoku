# テスト設計

## フレームワーク

- **Vitest** を使う（単体・統合）。E2E（Playwright 等）は当面導入しない。
- テストは対象ファイルに**コロケーション**し `*.test.ts` とする（例: `src/server/folders.test.ts`）。
- コマンドは `pnpm test`（`package.json` に追加。※依存の導入は別タスク）。

## 何をテストするか（優先順）

1. **ドメインロジック（最優先）** — `src/server/*` の純粋なルール。特に設計書の不変条件:
   - フォルダ移動の**循環参照チェック**（移動先が自分の子孫でないこと）
   - **未分類**判定（`document_folders` に1行も無い書類）
   - タグの **upsert**（既存は再利用・新規のみ作成）
2. **セキュリティ境界** — allowlist 判定（許可/非許可 email）、Zod スキーマの境界値（必須欠落・型不正で reject、正常系で通過）。
3. **統合（Route Handler）** — 「未認証 → 401」「不正入力 → 400」「正常 → 期待 JSON・ISO 8601 日時」の流れを確認。

## テストしやすい設計

- **DB / S3 非依存のロジックは純粋関数に切り出す**（循環参照チェックなど）。副作用と判断を分離してユニットで検証する。
- S3 はモックし、**署名 URL 発行の呼び出しパラメータ**（キー・有効期限・メソッド）を検証する。実際の PUT/GET はしない。
- DB を伴う統合テストが必要な場合はテスト用 Postgres を使い、テストごとにロールバック。既定はモック/純粋関数中心で軽く保つ。

## テストしないもの

- 画面の見た目・スタイル。
- Next.js / Drizzle / Auth.js など枠組み自体の挙動。

## 規約

- `describe` はテスト対象、`it` / `test` は「〜する」「〜のとき〜になる」を日本語で書いてよい。
- カバレッジ率は強制しない（家庭内・小規模）。**重要ロジックとセキュリティ境界を優先**する。
