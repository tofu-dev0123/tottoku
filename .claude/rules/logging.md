# ログ設計

## 方針

- Vercel のサーバレス環境では **stdout/stderr がそのままログ基盤に集約される**。**構造化 JSON を1行**で出す薄いロガーを `src/lib/logger.ts` に用意し、**必ず logger 経由で出力する**（`console.log` の直書きは避ける）。依存ライブラリは足さない。
- ログはサーバー側のみ。クライアント側に機微なログを残さない。

## 出力フォーマット

1行 = 1 JSON。最低限のフィールド:

| フィールド        | 内容                                                    |
| ----------------- | ------------------------------------------------------- |
| `level`           | `debug` / `info` / `warn` / `error`                     |
| `msg`             | 短い事象名（例 `"documents.create"`, `"unauthorized"`） |
| `time`            | ISO 8601                                                |
| `requestId`       | 可能なら付与（リクエスト単位の相関）                    |
| `userId`          | 操作主体（email ではなく user id）                      |
| （任意）`context` | 追加情報。**下記マスキング規則を厳守**                  |

- 本番は **info 以上**を出力。`debug` は開発時のみ。

## PII / 機微情報のマスキング（厳守）

家庭の書類が対象。**以下は絶対にログに出さない:**

- 書類の**実体・`title`・`memo`・ファイル名**（識別は `document.id` のみ）
- **S3 の署名付き URL・s3_key**、**IAM 認証情報**、**セッショントークン / Cookie**、**DATABASE_URL 等の秘密情報**
- email は原則 `userId` で代替。やむを得ず出す場合は**マスク**（例 `t***@example.com`）

## 何をログするか

- **API リクエスト**: メソッド・パス・ステータス・所要時間・`userId`（成功/失敗とも）。
- **エラー**: `level: error` でメッセージとスタックを出す（ただし上記の機微値はスタック/コンテキストから除外）。
- **S3 / DB 操作**: 値やキーではなく「操作種別 + 対象 id」（例 `msg: "s3.presign.issued", documentId`）。

## Route Handler での使い分け

- 想定内の失敗（401 未認証・400 バリデーション）は **`warn` / `info`**。
- 想定外の例外は **`error`** ログ + `{ error: string }` 応答。ユーザーには内部詳細を返さない。
